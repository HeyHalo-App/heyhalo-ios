# HaloiOS — running it for real

The iOS companion: text your Mac's Halo from your iPhone. The message rides
your own iCloud (Apple relays it), the thinking happens on your Mac, and nothing
passes through Halo's servers.

This is **slice B** (the phone client). The Mac side (slice A) is already merged.

## Build it on the simulator (no signing needed)

The project compiles + runs on the simulator with signing off. CloudKit can't
actually connect there (it needs the signed entitlement), so the app opens on
the welcome / onboarding screen and reports "iCloud isn't available right now"
with "Add this app to your iCloud to reach your Mac" — the honest could-not-
determine state, not a bug. (Onboarding shows on first launch and whenever the
phone isn't signed into iCloud; once signed in, "Start" drops into the chat.)

```sh
cd apps/HaloiOS
xcodebuild -project HaloiOS.xcodeproj -scheme HaloiOS \
  -sdk iphonesimulator -destination 'generic/platform=iOS Simulator' \
  -derivedDataPath .dd CODE_SIGNING_ALLOWED=NO build
```

Unit tests (the wire-protocol round-trip + the "matches the Mac" contract):

```sh
xcodebuild test -project HaloiOS.xcodeproj -scheme HaloiOS \
  -sdk iphonesimulator -destination 'platform=iOS Simulator,name=iPhone 17 Pro' \
  -derivedDataPath .dd CODE_SIGNING_ALLOWED=NO
```

## Make it actually text your Mac (the real run)

CloudKit needs a real, signed entitlement. Do this once in Xcode:

1. **Open the project.** `open apps/HaloiOS/HaloiOS.xcodeproj`.
2. **Set signing.** Select the `HaloiOS` target -> Signing & Capabilities ->
   set your **Team** (the same Apple Developer team the Mac app uses). Leave
   "Automatically manage signing" on.
3. **Add iCloud + CloudKit.** Still in Signing & Capabilities, click **+
   Capability** -> **iCloud**. Check **CloudKit**, then under Containers add /
   tick **`iCloud.com.silvercommerce.halo`** (the SAME container the Mac uses —
   this is what makes the two devices share one private database).
4. **Add Push Notifications.** **+ Capability** -> **Push Notifications**.
5. **Confirm Background Modes.** **+ Capability** -> **Background Modes** ->
   tick **Remote notifications**. (The `Info.plist` already lists
   `remote-notification`; the capability is what enrols the app for silent
   pushes.)
   - These three capabilities re-write `Resources/HaloiOS.entitlements`; the
     repo ships a starter version of that file with the right keys already in
     it, so step 3-5 mostly just attach your team's provisioning.
6. **Pick a destination signed into the right iCloud account.** Run on a
   physical iPhone, or a simulator, that is signed into the **SAME iCloud
   account as the Mac running Halo**. (Settings -> [your name] on the device;
   the Mac is signed into the same Apple ID.) This is the whole pairing step —
   there's no address to type, no QR code: same iCloud == same private database.
7. **On the Mac:** make sure Halo has the iCloud / CloudKit capability added
   (same container) and the `reachCloudKitEnabled` preference is **on**. With it
   on, the Mac's `ReachCloudKitHandler` ensures the `HaloReach` zone, subscribes
   for silent push, and answers inbound `user` messages.
8. **Run + text it.** Build & run HaloiOS on the device. Type a message and
   send. It writes a `role:user` record to the shared private DB; Apple relays
   it to the Mac; the Mac runs the agent turn and writes back a `role:halo`
   reply; Apple relays that to the phone, which renders it.

### Simulator with a real container (optional)

A signed simulator build (team + iCloud capability set, signing ON) can also
talk to CloudKit. Because the DEBUG simulator build skips container construction
by default (so the unsigned build stays launchable), pass the launch argument
**`-reachEnableCloudKit`** in the scheme's Run arguments to opt the signed
simulator build back in. On device and in Release this is automatic.

## Notes / things to verify with real signing

- **Anything CloudKit needs a signed entitlement.** The send/receive path, the
  zone subscription, the silent push, and the local-notification-on-reply were
  all written + compiled, but the live round-trip can only be exercised once the
  container capability is attached to a signing identity (steps above). The
  unit tests cover the wire format; they don't hit the network.
- **The connection indicator is honest about what the phone can see.** It can
  only verify its own iCloud account, so "Signed into iCloud" is as far as it
  goes on iCloud status alone — it never claims "Connected to your Mac" before a
  reply lands, because it can't see whether the Mac is awake or has the toggle
  on. The first real `halo` reply is the only end-to-end proof, and that
  promotes the header indicator to "Connected" (persisted, so a returning user
  isn't demoted). This needs a real provisioned device (or a signed simulator
  build with `-reachEnableCloudKit`) on the same iCloud account as the Mac, with
  the Mac's toggle on, to exercise end to end.
- **Mac silent-push reliability** to a non-foreground app is an open question
  (see the spec). The Mac handler polls every 30s as a fallback; the phone
  fetches on foreground + on push. If a push is slow, the reply still lands on
  the next foreground/fetch.
- **Approve / Deny** on a `needs-confirm` card are wired end to end: the in-app
  card and the Dynamic Island / Lock Screen buttons both write a `role:user`
  answer carrying the `confirmToken` to the private DB (`answerConfirm` /
  `ReachConfirmIntent`). Needs a real signed CloudKit path to exercise live.
- **Account gate / paywall** is live. Sign-in runs against `api.heyhalo.app`
  (GitHub OAuth, email magic link, or license key); the app reads plan state and
  unlocks Reach when the plan is active. The `PaywallView` never sells or links
  to a purchase (App Store multiplatform-service rule) — it only re-checks the
  account. An offline App Review demo account (`appreview@heyhalo.app`) shows the
  full flow with canned replies and no network.

## Regenerating the project

The `.xcodeproj` is generated from the files on disk. After adding/removing a
Swift file, re-run:

```sh
cd apps/HaloiOS && ruby generate_xcodeproj.rb
```
